# Архитектура DFIR AI Assistant

## Обзор

Система состоит из четырёх слоёв, каждый из которых можно развивать и тестировать независимо:

1. **Слой сбора/триажа** (input layer)
   - Триаж-пакеты (KAPE, Velociraptor, GRR, кастомные скрипты)
   - Полные образы дисков (E01, raw, vmdk, qcow2)
   - Отдельные артефакты (evtx, registry hives, prefetch, shimcache, browser history, plaso dumps)
   - Источник: локальная папка `data/triage/`, в перспективе - приёмник (uploader) с удалённых хостов

2. **Слой обработки (pipeline)**
   - Docker-контейнер с plaso/log2timeline
   - Прогон `log2timeline.py` -> `pinfo.py` -> `psort.py`
   - Выход: супертаймлайн в CSV/JSONL
   - Изоляция: контейнер одноразовый, evidence не модифицируется, читается read-only

3. **Слой знаний (RAG + turboVEC + LLM-wiki)**
   - Парсер таймлайна -> chunks (одно событие = один chunk)
   - Эмбеддер -> векторное хранилище turboVEC
   - LLM-wiki: парсер threat-intel отчётов -> статьи в формате, маппящемся на тот же RAG
   - Запрос к RAG возвращает релевантные события + релевантные статьи wiki одновременно

4. **Слой агентов**
   - Общий DFIR-агент: принимает описание инцидента, строит гипотезы, опрашивает RAG, формирует отчёт
   - Сабагенты (в перспективе): специализированы по доменам артефактов
     - registry-agent
     - evtx-agent
     - prefetch-agent
     - shimcache-agent
     - browser-agent
     - filesystem-agent
     - network-agent
     - threatintel-agent (поверх LLM-wiki)

```
       |                |
       v                v

       |

       |
       v

```

## Принципы проектирования

- **Один лог - один элемент RAG.** Грубое разбиение порождает галлюцинации (модель достраивает контекст между событиями). Чанкование по событию убирает этот класс ошибок: каждое извлечение - это цитата реального артефакта.
- **Evidence read-only.** Контейнер plaso монтирует evidence в read-only, в пайплайн передаётся только путь. Хэшируются исходники.
- **Изоляция обработки.** Каждый прогон plaso - отдельный контейнер, нет состояния между запусками.
- **RAG и wiki в едином формате.** И события таймлайна, и статьи wiki приводятся к одному schema (см. rag_design.md), что позволяет агенту делать гибридные запросы.
- **Переключаемый LLM-бэкенд.** Абстракция над провайдером (OpenAI-совместимый / Ollama / другой), чтобы можно было гонять локально и в облаке.
- **Аудитруемость.** Каждый ответ агента сопровождается списком source-событий и source-статей wiki, на которые он опирался. Нет цитаты - нет утверждения.

## Открытые вопросы

- Формат turboVEC: внутренний или обёртка над FAISS/Chroma (см. decisions.md)
- Стратегия обновления RAG при дообразовании evidence (инкремент vs. полный пересбор)
- Мульти-кейсовость: один turboVEC на кейс или общий с фильтром по case_id
- Язык взаимодействия с агентом: русский vs. английский vs. двуязычный

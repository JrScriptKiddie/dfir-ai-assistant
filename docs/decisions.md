# Журнал архитектурных решений

## ADR-001: Один лог - один элемент RAG

**Дата:** 2026-07-20
**Статус:** Accepted

**Контекст:** RAG обычно чанкует текст по фиксированному размеру окна с перекрытием. Для DFIR-таймлайна это плохо: события дискретны, между ними нет нарративной связи, модель достраивает контекст и галлюцинирует.

**Решение:** Чанковать по событию - один лог = один chunk. Текстом chunk служит человеко-читаемое представление события.

**Следствия:**
- Плюс: точность извлечения, цитируемость, нет галлюцинаций от достроенного контекста
- Плюс: metadata события хранится рядом, доступны фильтры
- Минус: больше chunks -> больше векторов -> больше памяти. Для миллиона событий ~300 МБ индекса, приемлемо.
- Минус: короткие события дают бедный эмбеддинг. Компенсируется тем, что text включает контекстные поля (host, user, source).

## ADR-002: plaso в Docker

**Дата:** 2026-07-20
**Статус:** Accepted

**Контекст:** plaso/log2timeline - Python-инструмент с большим количеством зависимостей, версия-зависимым поведением, потенциально нестабильный на разных ОС.

**Решение:** Запускать plaso в Docker-контейнере, монтируя evidence read-only.

**Следствия:**
- Плюс: воспроизводимость, изоляция, easy upgrade
- Плюс: безопасность - evidence не модифицируется
- Минус: зависимость от Docker, больше оверхед на запуск
- Альтернатива (отвергнута): установка plaso в venv - конфликтующие зависимости с RAG/LLM-стеком

## ADR-003: turboVEC - внутренний векторный стор

**Дата:** 2026-07-20
**Статус:** Proposed

**Контекст:** Готовые решения (Chroma, FAISS, Qdrant, Weaviate) имеют разный профиль: Chroma тяжёлый по зависимостям, FAISS - библиотека без API, Qdrant/Weaviate - серверные компоненты.

**Решение:** Сделать тонкую обёртку turboVEC поверх numpy + опционально FAISS. Один индекс = один файл-набор, без сервера, лёгкая сериализация.

**Следствия:**
- Плюс: минимум зависимостей, лёгкий перенос, предсказуемая производительность до ~10М векторов
- Плюс: легко бэкапить и версионировать
- Минус: нет распределённости, нет live-обновления с нескольких клиентов
- Альтернатива (запасная): если упрёмся в масштаб - миграция на Qdrant с тем же API

## ADR-004: LLM-wiki в том же RAG-формате

**Дата:** 2026-07-20
**Статус:** Accepted

**Контекст:** Хочется, чтобы агент опирался не только на свежие события кейса, но и на известные TTPs/IOCs из threat-intel отчётов.

**Решение:** Привести wiki-статьи к тому же chunk-формату, что и события, и индексировать в отдельный wiki-индекс turboVEC. Агент делает гибридные запросы.

**Следствия:**
- Плюс: единый API retrieval, минимальная duplicated-логика
- Плюс: в ответе hits помечены source=event|wiki, агент корректно их различает
- Минус: wiki-текст нарративен, чанкование по абзацам (а не по событию) - исключение из ADR-001, обосновано тем что wiki - не таймлайн

## ADR-005: Переключаемый LLM-бэкенд

**Дата:** 2026-07-20
**Статус:** Accepted

**Контекст:** DFIR-данные чувствительны, не всегда можно отправлять в облако. Нужна опция локального LLM.

**Решение:** Абстракция `LLMProvider` с реализациями: OpenAI-compatible, Ollama, (перспективно - vLLM).

**Следствия:**
- Плюс: конфиденциальность (локальный режим)
- Плюс: cost control
- Минус: разные модели дают разное качество, нужно тестировать
- Дефолт: Ollama с разумной моделью для приватности

## ADR-006: Сабагенты после стабилизации общего агента

**Дата:** 2026-07-20
**Статус:** Proposed

**Контекст:** Можно сразу строить мульти-агентную систему, но это усложнит отладку.

**Решение:** Сначала общий DFIR-агент с единым retrieval. После стабилизации - выделение сабагентов по доменам.

**Следствия:**
- Плюс: быстрее первый работающий результат
- Плюс: понятная эволюция
- Минус: общий агент может страдать от широты доменов, но это сигнал к выделению сабагента

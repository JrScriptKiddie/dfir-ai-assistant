# Дизайн RAG и turboVEC

## Принцип чанкования

**Один лог - один элемент RAG.**

Это ключевое архитектурное решение, направленное против галлюцинаций:
- При разбиении по фиксированному размеру окна модель вынуждена достраивать контекст между событиями, что приводит к галлюцинациям.
- При разбиении по событию каждое извлечение из RAG - это конкретный артефакт с конкретным timestamp. Агент отвечает цитатами, а не интерполяцией.

Формат chunk:

```json
{
  "id": "<sha1(case_id|timestamp|source|message)>",
  "text": "2024-03-12T08:14:22Z | EVTX | Security | 4624 | Logon | User: jsmith | LogonType: 3 | Workstation: WS01",
  "metadata": {
    "case_id": "uuid",
    "timestamp": "2024-03-12T08:14:22Z",
    "source": "EVTX",
    "sourcetype": "Security",
    "event_id": "4624",
    "parser": "winevtx",
    "host": "DC01",
    "user": "jsmith",
    "raw_hash": "<sha1 raw event>"
  }
}
```

`text` - человеко-читаемое представление, по нему строится эмбеддинг.
`metadata` - для фильтрации и аудита, не эмбеддится.

## turboVEC

turboVEC - компактное векторное хранилище для проекта.

Характеристики:
- Один индекс на кейс + опционально общий индекс wiki (см. ниже)
- Поддержка точечного поиска (top-k cosine) с фильтрами по metadata
- Персистентность: бинарный индекс + JSON-стор с metadata
- Лёгкая сериализация для бэкапа/переноса между машинами

Дизайн (первый вариант):
- Внутреннее хранилище: numpy-матрица векторов + dict id->metadata
- Опционально: FAISS IndexFlatIP под капотом для скорости на больших кейсах
- API:
  - `add(chunks: list[Chunk], embeddings: list[Vector])`
  - `query(text: str, k: int, filters: dict) -> list[Hit]`
  - `save(path)`, `load(path)`
  - `stats() -> {n_vectors, dim, case_id, created_at}`

Имя "turboVEC" отражает идею: лёгкий, быстрый, без тяжёлой СУБД-зависимости, но достаточный для DFIR-объёмов (миллионы событий на кейс).

## Гибридный запрос: события + wiki

RAG возвращает два параллельных потока:
1. Hits по событиям таймлайна кейса
2. Hits по статьям LLM-wiki (threat-intel)

Это позволяет агенту сопоставить свежее событие с known-TTP/IOC из публичных отчётов. Формат wiki-статей мапится на тот же chunk-формат (см. llm_wiki.md).

## Фильтры

Агент может фильтровать RAG по:
- time_range: [start, end]
- source / sourcetype / event_id
- host / user
- parser
- case_id (для мульти-кейс конфигураций)

Фильтры применяются post-retrieval (на metadata), что упрощает индекс.

## Аудит и цитируемость

Каждый hit возвращает:
- score (cosine)
- chunk.id
- chunk.text
- chunk.metadata

Агент обязан в ответе ссылаться на chunk.id. UI/CLI рендерит источники отдельным блоком. Без источника утверждение считается гипотезой, не фактом.

## Эмбеддинг-модель

Переключаемая:
- `nomic-embed-text` (Ollama, локально) - дефолт для приватности
- `text-embedding-3-small` (OpenAI) - если допустима отправка в облако
- `bge-small-en-v1.5` / `multilingual-e5-small` - альтернативы

Размерность вектора определяется моделью, turboVEC хранит dim в метаданных индекса.
